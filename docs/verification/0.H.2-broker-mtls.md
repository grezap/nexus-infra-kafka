This document is the 0.H.2 exit-gate proof: both KRaft clusters
(`kafka-east` + `kafka-west`) are flipped from the 0.H.1 PLAINTEXT
backplane to **mutual TLS** — per-node Vault PKI leaf certs, SSL on the
client *and* controller listeners, `ssl.client.auth=required` everywhere
— and each cluster still elects a controller quorum and passes an RF=3
produce/consume round-trip, all verified over mTLS.

## Design

mTLS on the existing 0.H.1 ports — only the wire protocol flips, so
`role-overlay-nftables-backplane.tf` needs no change:

| Listener | 0.H.1 | 0.H.2 |
|---|---|---|
| client + inter-broker (`:9092`) | `PLAINTEXT` | `SSL`, `inter.broker.listener.name=SSL` |
| controller quorum (`:9093`) | `CONTROLLER`→`PLAINTEXT` | `CONTROLLER`→`SSL` |

`ssl.client.auth=required` makes it **mutual** TLS everywhere —
inter-broker replication, the controller quorum, *and* external clients
all present a client cert. There is no "operator API without a client
cert" carve-out: every Kafka client in the lab runs *from* a broker and
uses that broker's own keystore as its client identity (the
`kafka-broker` PKI role has `client_flag=true`). The dropped
`/etc/nexus-kafka/client-ssl.properties` points the CLI tools at exactly
that keystore.

Keystore format: Kafka 3.8's native PEM type (`ssl.keystore.type=PEM`) —
no JKS, no `keytool`, no keystore password. The Vault Agent renders one
`bundle.pem` (leaf + key + issuing CA); the `kafka-tls-split.sh`
post-render command splits it into `keystore.pem` (key + leaf +
intermediate, full chain) and `truststore.pem` (the intermediate CA).

## Step 2 — the PKCS#1 → PKCS#8 fix

The v1 apply rendered all 6 certs cleanly (Phase 1) but every broker
crash-looped on the Phase 2 restart:

```
Caused by: java.security.InvalidKeyException: Unable to decode key
Caused by: java.io.IOException: algid parse error, not a sequence
        at java.base/sun.security.pkcs.PKCS8Key.decode(PKCS8Key.java:112)
```

**Root cause:** Vault PKI issues RSA private keys in **PKCS#1**
(`-----BEGIN RSA PRIVATE KEY-----`). Kafka's PEM keystore parser is Java,
which only accepts **PKCS#8** (`-----BEGIN PRIVATE KEY-----`). Consul and
Nomad never hit this — Go's `crypto/tls` reads both formats.

**Fix** (`kafka_tls_v` v1→v2): `kafka-tls-split.sh` now runs the key
through `openssl pkcs8 -topk8 -nocrypt` before assembling `keystore.pem`
(idempotent — `-topk8` on an already-PKCS#8 key re-emits it). Also added
`systemctl reset-failed` ahead of the Phase 2 restart so a re-run after a
start-limited prior attempt isn't blocked. The re-apply was a clean
terraform taint-recovery (`1 added, 1 destroyed`): the destroy
provisioner restored PLAINTEXT, then the create provisioner re-ran the
full flip — the create won, end state is mTLS.

The 6 `kafka_vault_agent` resources (one `nexus-vault-agent.service` per

can't elect during the gap (the consul-tls / nomad-tls lesson; KRaft's
controller quorum is a Raft group exactly like Consul/Nomad's). The two
clusters flip one after the other (west stays up while east converges).

## What's proven

1. **Both KRaft clusters run on mutual TLS.** Inter-broker replication,
   the controller quorum, and external clients all require + present a
   client cert. No PLAINTEXT listener remains.
2. **Per-node Vault PKI identity.** Each broker's `nexus-vault-agent.service`
   issues + auto-renews its own leaf cert from `pki_int/issue/kafka-broker`
   via a narrow per-host AppRole — no shared key, no static secret.
3. **The flip is non-destructive.** KRaft cluster IDs + node IDs are
   preserved; only the listener wire protocol changed.
4. **The parallel big-bang restart is the right primitive.** Per-cluster,
   parallel-within-cluster — the controller quorum re-formed over SSL on
   both clusters with no no-leader gap.
5. **Selective + idempotent.** `enable_kafka_tls` / `enable_kafka_vault_agents`
   gate the layer; re-applying with the clusters already on mTLS is a
   no-op-fast. The destroy provisioner restores a working PLAINTEXT state.

## What got fixed during the run

1. **PKCS#1 → PKCS#8 key conversion** — see Step 2. `kafka-tls-split.sh`
   now converts the Vault-PKI RSA key to PKCS#8 via `openssl pkcs8
   -topk8`. (`kafka_tls_v` v2)
2. **`systemctl reset-failed` before the Phase 2 restart** — so a re-run
   after a start-limited prior attempt (`StartLimitBurst=15`) isn't
   blocked. (`kafka_tls_v` v2)

## What's deferred to 0.H.3+

1. **Kafka ACLs / SASL** — 0.H.2 is mutual TLS only (channel encryption +
   peer identity), not authorization. Topic/operation ACLs are a later
   concern.
2. **The ecosystem nodes** — schema-registry / kafka-connect / ksqldb /
   mm2 / kafka-rest are not cloned yet. 0.H.3-0.H.5 bring them up and
   they consume the brokers' mTLS listeners.

## Reverting on failure

`terraform destroy`-targeting `null_resource.kafka_tls` runs the destroy
provisioner: it restores the PLAINTEXT `server.properties` (frozen into
triggers), removes the TLS artifacts + Vault Agent template, and restarts
`nexus-vault-agent` + `kafka` — leaving both clusters on a working 0.H.1
PLAINTEXT state. The Vault-side PKI role + AppRoles in the security env
are untouched (idempotent; harmless if the kafka env is reverted).
