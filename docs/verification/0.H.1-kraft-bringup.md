## Packer build — `kafka-node` template

Built from `packer/kafka-node/kafka-node.pkr.hcl`. Build v1 failed at
"Download + verify + extract Apache Kafka":

```
sha512sum: 'standard input': no properly formatted checksum lines found
```

Root cause: Apache's `kafka_2.13-3.8.1.tgz.sha512` sidecar wraps the hash
across indented multi-line continuation lines; the single-line
`awk -F': '` parse truncated it. Fix: pin the literal lowercased +
concatenated sha512 in `kafka_node_kafka_sha512` and verify via
`echo "<hash>  <file>" | sha512sum -c -`. Committed in
[`3a59928`](https://github.com/grezap/nexus-infra-kafka/commit/3a59928).
Build v2 succeeded — all role systemd units delivered **disabled**.

## What's proven

1. **Two independent KRaft clusters are live.** `kafka-east` (primary) and
   `kafka-west` (DR) each run 3 nodes in combined broker+controller mode,
   each with an elected controller quorum of 3 voters.
2. **The `kafka-node` template is cold-rebuild-complete.** One
   parameterised Debian 13 template clones into all 6 brokers; the same
   template carries the Confluent Community bits for 0.H.3-0.H.5.
   firstboot does NIC/hostname/identity only — KRaft formatting is
   Terraform-time because it needs a per-cluster UUID.
3. **Selective apply works.** `enable_kafka_east` / `enable_kafka_west` /
   per-VM toggles gate every `module.vm` block; `count = ... ? 1 : 0`
   resources default to the steady-state `true` per
   `memory/feedback_terraform_partial_apply_destroys_resources.md`.
4. **RF=3 is real.** A produce → consume round-trip on a fresh
   3-partition / RF=3 topic succeeds on each cluster, and
   `kafka-topics --describe` confirms 3 replicas per partition.
5. **The VMnet10 backplane carries the controller quorum.** Brokers
   advertise the CONTROLLER listener on their VMnet10 IP; the nftables
   overlay opens the segment. PLAINTEXT in 0.H.1; mTLS lands in 0.H.2.

## What got fixed during the run

1. **Apache Kafka sha512 multi-line sidecar parse** — see "Packer build"
   above; pinned the literal hash. `3a59928`.
2. **Commit-message backtick command substitution** — a `git commit -m`
   message containing backticks around shell snippets caused bash to
   execute `awk` / `sha512sum`, mangling the message. Fixed via
   `git commit --amend -F -` with a heredoc (no backticks).
3. **nftables overlay leftover buggy line** — a stray malformed
   `bash -s -- '$remote'` line preceding the correct version was removed.

## What's deferred to 0.H.2+

1. **Broker mTLS** — 0.H.1 brokers run PLAINTEXT on both the client
   (9092) and controller (9093) listeners. 0.H.2 adds a Vault PKI
   `kafka-broker` role, per-node Vault Agents, and an SSL listener +
   inter-broker + controller-quorum mTLS.
2. **The ecosystem nodes** — schema-registry / kafka-connect / ksqldb /
   mm2 / kafka-rest VMs are not cloned in 0.H.1 (their dnsmasq
   reservations are live and idle, ready). 0.H.3-0.H.5 bring them up.
3. **`vms.yaml` ksqldb-2 IP typo** — line 109's `.99` should be `.98`;
   the firstboot map and dnsmasq reservation already use `.98`. The
   canon file gets corrected in the 0.H.6 close-out batch.

## Reverting on failure

`pwsh -File scripts/kafka.ps1 destroy` tears down the kafka env (6
clones + 4 overlays); each overlay's destroy provisioner is idempotent.
The foundation kafka-reservations overlay is removed by re-applying
foundation with `enable_kafka_dhcp_reservations=false` (or
`terraform destroy`-targeting that resource). Vault KV state is
untouched — 0.H.1 is PLAINTEXT and stores no secrets.
