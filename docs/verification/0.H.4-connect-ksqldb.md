## Design

The 4 nodes are Kafka **clients** of the `kafka-east` cluster + serve
their own HTTPS operator listeners. They reuse `role-overlay-ecosystem-tls.tf`
for their Vault-PKI keystores. New in 0.H.4: that overlay now also emits a
**PKCS#12** keystore/truststore beside the PEM pair (see "What got fixed").

| Surface | TLS posture |
|---|---|
| Connect / ksqlDB ŌåÆ `kafka-east` brokers | mutual TLS (the brokers require client certs) |
| Connect REST `:8083` | HTTPS, **PKCS#12** keystore (KIP-208 `listeners.https.ssl.*`), `client.auth=none` |
| ksqlDB REST `:8088` + ksqlDBŌåÆSR client | HTTPS, **PKCS#12** (bare `ssl.*` + `ksql.schema.registry.ssl.*`), `client.authentication=NONE` |
| Connect Kafka client | PEM (`producer.`/`consumer.`/`admin.` prefixed -- top-level `ssl.*` doesn't cascade) |
| ksqlDB Kafka client | PKCS#12 (bare `ssl.*` is inherited by the embedded Kafka clients -- so it must match the REST listener's format) |

Overlays (each independently `-target`-able):
- `role-overlay-connect.tf` ŌĆö installs the Debezium plugins
  (`/opt/connect-plugins/debezium-connector-{postgres,sqlserver}`,
  version `2.7.3.Final`), renders `connect-distributed.properties`,
  sequential 2-worker start, verifies `/connector-plugins`.
- `role-overlay-ksqldb.tf` ŌĆö renders `ksqldb-server.properties`,
  sequential 2-node start, verifies the `/info` agreement +
  `SHOW TOPICS;`.

Connect + ksqlDB self-create their internal/command topics via AdminClient
(works with the brokers' `auto.create.topics.enable=false`; no Kafka ACLs);
the pairs start **sequentially** so node-1 creates the topics and node-2
joins.

## Step 2 ŌĆö `terraform apply` (8 attempts) ŌĆö the bug chain

The Connect + ksqlDB bring-up surfaced a chain of issues, each fixed in
turn. They are recorded here in full because several are reusable lessons
for the JVM ecosystem tier.

1. **id-trigger cascade (0.H.1 overlays).** Bumping the nftables overlay's
   version changed `kafka_broker_config`'s `nftables_id` trigger ŌåÆ
   `broker_config` re-ran ŌåÆ `kraft_format` ŌåÆ `broker_start_verify` (chained
   on each other's resource ids). `broker_config` re-rendered **PLAINTEXT**
   `server.properties` over the 0.H.2 SSL config, and `broker_start_verify`'s
   PLAINTEXT quorum probe **OOM'd** against the now-mTLS brokers (a PLAINTEXT
   Kafka client reads the TLS handshake bytes as a message length and tries
   to allocate gigabytes of heap). 0.H.3 only survived this by luck ŌĆö its
   `kafka_tls` recreate happened to flip the brokers back to PLAINTEXT
   first. **Fix:** dropped the `nftables_id`/`config_id`/`format_id`
   id-triggers (the `va_ids` lesson again); `broker_config` now skips any
   broker that already has `/etc/nexus-kafka/client-ssl.properties` (the
   reliable TLS-flip marker); `broker_start_verify` detects that same
   marker and probes with `--command-config` when the broker is mTLS;
   `kafka_tls_v`ŌåÆ4 re-asserted the SSL `server.properties`.
2. **Connect + ksqlDB REST listeners reject `ssl.keystore.type=PEM`.**
   Apache Kafka's Connect `RestServer` passes the keystore type straight
   to Jetty's `KeyStore.getInstance()` (`NoSuchAlgorithmException: PEM
   KeyStore not available`); ksqlDB's `KsqlRestConfig` validates against
   `{JKS, PKCS12, BCFKS}` and rejects `PEM`. (Schema Registry / REST Proxy
   accept PEM only because they use Confluent `rest-utils` with its
   PEMŌåÆkeystore helper.) **Fix:** `role-overlay-ecosystem-tls.tf` now also
   emits `keystore.p12` + `truststore.p12`; the Connect/ksqlDB REST
   listeners use PKCS#12.
3. **ksqlDB UDF SecurityManager on Java 21.** ksqlDB calls
   `System.setSecurityManager()`, which throws `UnsupportedOperationException`
   on Java 17+ (the kafka-node template runs Temurin JDK 21). **Fix:**
   `ksql.udf.enable.security.manager=false`.
4. **ksqlDB bare `ssl.*` bleed.** ksqlDB's bare `ssl.*` (the REST listener,
   forced to PKCS#12) is inherited by its embedded Kafka clients; a
   keystore password is invalid with a PEM keystore. **Fix:** ksqlDB uses
   **PKCS#12 everywhere** (Kafka client + REST listener + SR client).
5. **Empty `truststore.p12`.** `openssl pkcs12 -export -nokeys` produced a
   cert bag Java loads as empty (`trustAnchors parameter must be
   non-empty`). **Fix:** build the truststore with `keytool -importcert`
   (a proper `trustedCertEntry`).
6. **ksqlDB AdminClient PLAINTEXT.** ksqlDB's `ServiceContext` AdminClient
   (startup `getKafkaClusterId`) reads the **bare** client config, which
   had `ssl.*` but no bare `security.protocol` ŌåÆ defaulted to PLAINTEXT ŌåÆ
   the mTLS brokers disconnected it. **Fix:** added bare
   `security.protocol=SSL`.
7. **`/clusterStatus` cluster check.** ksqlDB's `/clusterStatus` is
   heartbeat-driven and only lists *peers* once a persistent query exists
   (peer discovery rides Kafka Streams group metadata) ŌĆö a fresh,
   query-less cluster always shows just the local node. **Fix:** the
   cluster-membership verification keys on the deterministic `/info`
   agreement (both nodes `RUNNING` on the same `ksql.service.id` + the same
   `kafkaClusterId`) instead. `ksql.heartbeat.enable=true` is still set for
   real pull-query routing once queries land.

## What's proven

1. **The Kafka Connect distributed cluster is live.** Two workers share
   `group.id=nexus-connect-east`, connect to `kafka-east` over mutual TLS,
   self-created the 3 internal topics, and serve HTTPS REST on `:8083`.
2. **Debezium is loaded.** The Postgres + SQL Server connector classes are
   registered (`/connector-plugins`) ŌĆö ready for CDC sources in a later
   data-tier phase (no source DB exists yet, so no connector is
   configured).
3. **The ksqlDB cluster is live.** Two nodes share
   `ksql.service.id=nexus_ksql_east_` + the one command topic, connect to
   `kafka-east` over mutual TLS, are wired to the Schema Registry HA pair,
   and serve HTTPS REST on `:8088`. A `SHOW TOPICS;` round-trip works.
4. **Per-node Vault-PKI identity for 13 of 15 tier nodes.** Each ecosystem
   node runs its own `nexus-vault-agent.service` issuing a leaf cert from
   `pki_int/issue/kafka-broker`.
5. **The id-trigger cascade bug is sealed.** The 0.H.1 broker overlays no
   longer chain on resource ids ŌĆö re-applies in 0.H.5+ won't churn the
   brokers.

## What got fixed during the run

See "Step 2 ŌĆö the bug chain" above ŌĆö seven issues, all fixed in the
overlays:
1. id-trigger cascade in the 0.H.1 broker overlays (`broker_config` v2,
   `kraft_format` v2 ŌĆö id-triggers dropped, TLS-mode-aware).
2. Connect + ksqlDB REST listeners need PKCS#12, not PEM
   (`ecosystem_tls_v` v2).
3. ksqlDB UDF SecurityManager disabled for Java 21.
4. ksqlDB all-PKCS#12 (bare `ssl.*` bleed).
5. `truststore.p12` via `keytool` not `openssl -nokeys`
   (`ecosystem_tls_v` v3).
6. ksqlDB bare `security.protocol=SSL` for the AdminClient.
7. ksqlDB cluster verification via `/info` agreement, not `/clusterStatus`.

## Reverting on failure

`terraform destroy`-targeting `null_resource.kafka_connect` /
`null_resource.kafka_ksqldb` stops + disables the respective services and
removes the rendered config (the `/opt/connect-plugins` Debezium JARs are
left in place ŌĆö harmless, idempotent). The 4 ecosystem VM clones are
removed by `terraform destroy` of their `module.vm` blocks. The brokers +
the 0.H.3 ecosystem nodes + the Vault-side state are untouched.
