## Design

The 2 MM2 nodes are Kafka **clients of both** KRaft clusters. They reuse
`role-overlay-ecosystem-tls.tf` for their Vault-PKI keystores. Each runs
Apache Kafka's `connect-mirror-maker.sh` in **dedicated mode** — MM2
ships inside Apache Kafka, no Confluent component.

**Topology — one flow per node, deterministic:**

| Node | Flow | `--clusters` | Internal topics land in |
|---|---|---|---|
| `mm2-1` (.85) | `east→west` | `west` | `west` |
| `mm2-2` (.86) | `west→east` | `east` | `east` |

The two nodes do **not** cluster with each other — each owns exactly one
replication flow. `mm2.properties` is the common base (both clusters
registered, per-cluster mTLS) with only that node's one
`<src>→<dst>.enabled = true` line; a systemd drop-in
(`/etc/systemd/system/mm2.service.d/10-clusters.conf`) appends
`--clusters <target>` to the baked unit's `ExecStart` (belt + suspenders
— the `enabled=` line alone already pins the direction).

**TLS posture** (research-confirmed for Apache Kafka 3.8):

| Surface | TLS posture |
|---|---|
| MM2 → `kafka-east` + `kafka-west` brokers | mutual TLS (both clusters require client certs) |
| MM2 producer / consumer / admin clients | **PEM** — in MM2 dedicated mode the cluster-level `<alias>.ssl.*` / `<alias>.security.protocol` **auto-cascades** to all three client types (`MirrorMakerConfig.clusterProps()` does `putIfAbsent("producer."+k,...)`); the SSL block is written **once per alias** |
| Embedded Connect REST | **off** — `dedicated.mode.enable.internal.rest` defaults to `false`; it is only needed for multi-node MM2 coordination, which this one-node-per-flow topology does not have |

Because the embedded REST server is never stood up, the
Apache-Kafka-`RestServer`-rejects-PEM problem (0.H.4's Connect/ksqlDB
pain) **never arises** here — the `.p12` files on the MM2 nodes are
simply unused.

`DefaultReplicationPolicy`: a topic `T` on `east` is mirrored to `west`
as `east.T` (source-alias prefix). The prefix is what keeps the
bidirectional pair loop-safe — `east.`-prefixed topics are recognised as
already-remote and not re-mirrored back.

Overlay: `role-overlay-mm2.tf` — renders `mm2.properties` + the
`--clusters` drop-in per node, sequential start, per-node verify (journal
sanity → `heartbeats` topic on the target → the exit-gate round-trip).

## Step 1 — Vault-side state

The 2 MM2 nodes' policies + AppRoles were added to the security env
(`kafka_policies_overlay_v` v4, `kafka_approles_v` v4) — **15 kafka-node
Vault Agents total, the whole tier**. The MM2 policy `cluster = "both"`
(it spans both clusters); the policy body is the same minimal PKI-leaf +
token-self-management shape as the rest of the tier. The `kafka-broker`
PKI role already covered all 15 tier hostnames (0.H.3), so it needed no
change.

## Step 2 — `terraform apply` (2 attempts)

**Attempt 1 failed — but MM2 itself was healthy.** The `kafka_mm2`
overlay's verification probe ran the Kafka CLI tools
(`kafka-topics.sh --list`) as `nexusadmin` *without* `sudo`. The CLI
threw `java.nio.file.AccessDeniedException: /etc/nexus-kafka/client-ssl.properties`
— `/etc/nexus-kafka/` is `0750 root:kafka` and `nexusadmin` is not in the
`kafka` group, so it cannot **traverse** the directory to read the client
config or the keystore it points at. This is the
`feedback_sudo_required_for_consul_etc_traverse.md` lesson (the Consul
`/etc/consul.d/` 0750 case), Kafka edition. The MM2 journal at the time
already showed all three connectors (`MirrorSourceConnector`,
`MirrorCheckpointConnector`, `MirrorHeartbeatConnector`) committing
acknowledged offsets — mirroring was working; only the probe was blind.

**Fix:** every Kafka CLI invocation in `role-overlay-mm2.tf` and
`smoke-0.H.5.ps1` now runs under `sudo`. Also hardened the topic-list
match: `(?m)^heartbeats$` → `(?m)^heartbeats\r?$`, since `Out-String`
rejoins ssh output with `\r\n` and .NET `$` would not match before the
`\r`.

**Attempt 2 — all green.** Both exit-gate round-trips passed:

## What's proven

1. **The MirrorMaker 2 DR pair is live.** `mm2-1` mirrors `kafka-east →
   kafka-west`, `mm2-2` mirrors `kafka-west → kafka-east` — each running
   `connect-mirror-maker.sh` in dedicated mode, one flow per node.
2. **The Phase 0.H exit gate is cleared.** A fresh record produced to
   `dr-gate-test` on `kafka-east` appears on `east.dr-gate-test` on
   `kafka-west` (and the reverse) — verified twice, once inline by the
   overlay and once independently by the smoke gate.
3. **Cross-cluster traffic is mutual-TLS.** MM2's producer/consumer/admin
   clients authenticate into both clusters with the node's Vault-PKI leaf
   cert; the journal shows zero TLS/auth failures.
4. **Per-node Vault-PKI identity for all 15 of 15 tier nodes.** Every
   `03-kafka` node now runs its own `nexus-vault-agent.service` issuing a
   leaf cert from `pki_int/issue/kafka-broker`.
5. **The whole `03-kafka` tier is up** — 6 KRaft brokers + Schema
   Registry ×2 + REST Proxy + Connect ×2 + ksqlDB ×2 + MM2 ×2 = 15 VMs.

## What got fixed during the run

1. **Kafka CLI probes need `sudo`** — `/etc/nexus-kafka/` is `0750
   root:kafka`, so `nexusadmin` cannot traverse it to read
   `client-ssl.properties` / the keystore. Every Kafka CLI call in
   `role-overlay-mm2.tf` + `smoke-0.H.5.ps1` now runs under `sudo` (the
   `feedback_sudo_required_for_consul_etc_traverse.md` lesson, Kafka
   edition).
2. **`Out-String` `\r\n` vs `(?m)$`** — the topic-list exact-match regex
   is now `\r?`-tolerant.

## Reverting on failure

`terraform destroy`-targeting `null_resource.kafka_mm2` stops + disables
`mm2.service` and removes `mm2.properties` + the `--clusters` drop-in on
both MM2 nodes (idempotent). The 2 MM2 VM clones are removed by
`terraform destroy` of their `module.mm2_*` blocks. The mirrored topics
(`east.*` on west, `west.*` on east) and MM2's internal topics persist on
the brokers until manually deleted — harmless. The brokers + the 0.H.3 /
0.H.4 ecosystem nodes + the Vault-side state are untouched.
