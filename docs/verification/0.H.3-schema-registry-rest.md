## Design

The 3 ecosystem nodes are Kafka **clients** of the `kafka-east` cluster.
Each connects to the brokers over mutual TLS (the brokers run
`ssl.client.auth=required`), so each holds its own Vault-PKI keystore;
each also serves its own HTTPS listener from the same cert. Confluent
7.7.1's Jetty REST listeners accept `ssl.keystore.type=PEM` (verified in
the `rest-utils` source), so SR + REST use the **same password-less PEM
keystore** as the brokers — no JKS, no `keytool`, no keystore password.

| Surface | TLS posture |
|---|---|
| SR / REST → `kafka-east` brokers (`kafkastore.ssl.*` / `client.ssl.*`) | **mutual TLS** — the brokers require client certs |
| SR `:8081` + REST `:8082` own HTTPS listeners | server-side TLS, `ssl.client.authentication=NONE` — the consul-tls operator-API precedent (`https.verify_incoming=false`) |
| SR HA pair inter-instance leader-forwarding | HTTPS (`inter.instance.protocol=https`), same keystore |

Overlay structure (each independently `-target`-able):
- `role-overlay-ecosystem-tls.tf` — renders the PEM keystore/truststore +
  `client-ssl.properties` on each ecosystem node (the ecosystem-node
  analogue of `role-overlay-kafka-tls.tf`'s Phase 1). Per-node, sequential
  — no service coordination.
- `role-overlay-schema-registry.tf` — pre-creates `_schemas`, renders
  `schema-registry.properties` per node, starts the HA pair, HA-verifies.
- `role-overlay-rest.tf` — renders `kafka-rest.properties`, starts the
  REST Proxy, `/topics`-verifies.

## Step 2 — `terraform apply` (kafka env)

The v1 apply rendered everything correctly — `_schemas` pre-created, both
SR nodes started, both HTTPS listeners live — but the overlay's own
HA-check step failed on an over-escaped JSON payload. The fix: the SR API
body is `{"schema": "<schema as a JSON string>"}` — the outer object uses
plain quotes; only the *inner* schema-string quotes are `\"`-escaped. The
v1 code had `\"` on the outer quotes too. Fixed in
`role-overlay-schema-registry.tf` (and the same bug in two
`smoke-0.H.3.ps1` payloads).

### The 0.H.2 churn-fix (applied in this sub-phase)

Reviewing the re-apply behaviour surfaced a latent 0.H.2 coupling bug:
`kafka_vault_agent`'s destroy provisioner did `rm -rf /etc/vault-agent/`
(wiping the `60-template-kafka-tls.hcl` that the TLS overlays own), and
`kafka_tls`'s `va_ids` trigger re-ran the whole broker mTLS flip whenever
*any* Vault Agent id changed — which it does on every security-env apply
(secret-id rotation). Left unfixed, every 0.H.x apply cycle would
needlessly restart both broker clusters. Fixed:

- `kafka_vault_agent` destroy is now **surgical** — removes only the
  files it installs (`00-base.hcl` + creds + unit), leaving the TLS
  template intact.
- `kafka_tls` + `kafka_ecosystem_tls` **dropped the `va_ids` trigger** —
  they re-run only when the broker/node *set* changes. (`kafka_tls_v` v3.)

The 0.H.3 apply pays a **one-time** `kafka_tls` recreate (its triggers
changed) — the brokers restarted once, cleanly, and re-verified on mTLS
(`[kafka-tls] OK -- both KRaft clusters on mutual TLS`). After this,
`kafka_tls` is stable; future 0.H.x applies will not touch the brokers.

## Step 3 — `smoke-0.H.3.ps1` summary

cluster was healthy throughout):
1. the `_schemas`-describe probe ran `kafka-topics --bootstrap-server

## What's proven

1. **The Schema Registry HA pair is live.** Both nodes back the
   `_schemas` topic on `kafka-east` over mTLS and elect a leader via
   Kafka group coordination (no ZooKeeper). A schema registered on
   `schema-registry-1` is immediately readable from `schema-registry-2`.
2. **The REST Proxy is live.** `kafka-rest-1` bridges HTTP → Kafka over
   mTLS and is wired to the SR HA pair for schema-aware formats; a JSON
   produce/consume round-trip succeeds over HTTPS.
3. **Per-node Vault-PKI identity for the ecosystem tier.** Each ecosystem
   node runs its own `nexus-vault-agent.service` issuing a leaf cert from
   `pki_int/issue/kafka-broker` via a narrow per-host AppRole — the same
   pattern as the brokers, now spanning 9 of the 15 tier nodes.
4. **`role-overlay-ecosystem-tls.tf` is reusable.** It renders the PEM
   keystore on *any* ecosystem node — 0.H.4 (kafka-connect, ksqldb) and
   0.H.5 (mm2) extend its node list with no new TLS code.
5. **Re-applies no longer churn the brokers.** The 0.H.2 coupling bug is
   fixed; `kafka_tls` is keyed on the broker set, not on Vault Agent ids.

## What's deferred to 0.H.4+

1. **The remaining 6 ecosystem nodes** — kafka-connect ×2 + ksqldb ×2
   (0.H.4), mm2 ×2 (0.H.5). They reuse `role-overlay-ecosystem-tls.tf`
   and the security env's per-host AppRole pattern.
2. **Kafka ACLs / SASL** — still deferred (0.H.x is mTLS, not authz).
3. **Schema Registry / REST listener client-cert auth** — the operator
   HTTPS listeners are server-side TLS only (`ssl.client.authentication=NONE`),
   matching the consul-tls operator-API precedent.

## Reverting on failure

`terraform destroy`-targeting `null_resource.kafka_rest` /
`null_resource.kafka_schema_registry` stops + disables the respective
services and removes the rendered config. `null_resource.kafka_ecosystem_tls`'s
destroy removes the cert artifacts + Vault Agent template. The 3 ecosystem
VM clones are removed by `terraform destroy` of their `module.vm` blocks.
The brokers + the Vault-side PKI/AppRole state are untouched.
